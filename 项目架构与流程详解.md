## 1. 项目概述

### 1.1 什么是 Easy-Chat？

Easy-Chat 是一个基于 Go-Zero 微服务框架开发的即时通讯系统（类似微信、QQ）。

**核心功能**：
- 用户注册、登录
- 好友管理（添加好友、好友列表）
- 群组管理（创建群、加入群、群成员管理）
- 即时通讯（单聊、群聊）
- 消息推送（实时消息通知）

### 1.2 为什么使用微服务架构？

**传统单体应用的问题**：
```
┌─────────────────────────────────┐
│      单体应用（Monolith）        │
│  ┌──────────────────────────┐  │
│  │  用户模块                 │  │
│  │  好友模块                 │  │  ← 所有功能在一个应用里
│  │  群组模块                 │  │
│  │  消息模块                 │  │
│  └──────────────────────────┘  │
└─────────────────────────────────┘
```

**问题**：
- 代码耦合严重，改一个功能可能影响其他功能
- 无法独立扩展（比如只想扩展消息模块，但必须扩展整个应用）
- 部署困难（一个小改动需要重新部署整个应用）

**微服务架构的优势**：
```
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│ 用户服务  │  │ 社交服务  │  │ IM服务   │  │ 任务服务  │
│ (User)   │  │ (Social) │  │ (IM)     │  │ (Task)   │
└──────────┘  └──────────┘  └──────────┘  └──────────┘
     ↓             ↓             ↓             ↓
  独立部署      独立扩展      独立开发      独立维护
```

**优势**：
- 服务独立：每个服务可以独立开发、部署、扩展
- 技术灵活：不同服务可以使用不同的技术栈
- 故障隔离：一个服务出问题不会影响其他服务
- 团队协作：不同团队可以负责不同的服务

### 2.2 架构分层说明

#### 第一层：接入层（API Gateway）

**作用**：统一入口，所有外部请求都先到达这里

**组件**：APISIX（Apache APISIX）

**功能**：
1. **路由转发**：根据请求路径转发到对应的服务
   ```
   /api/user/*    → User API 服务
   /api/social/*  → Social API 服务
   /api/im/*      → IM API 服务
   ```

2. **负载均衡**：当一个服务有多个实例时，自动分配请求
   ```
   请求 → APISIX → [User API 实例1]
                 → [User API 实例2]  ← 自动选择
                 → [User API 实例3]
   ```

3. **认证鉴权**：检查用户是否登录，是否有权限
   ```
   请求 → 检查 Token → 有效 → 放行
                    → 无效 → 返回 401 未授权
   ```

4. **限流熔断**：防止服务被压垮
   ```
   正常情况：请求 → 服务
   流量过大：请求 → 限流 → 返回 429 Too Many Requests
   服务异常：请求 → 熔断 → 返回 503 Service Unavailable
   ```

#### 第二层：API 层（HTTP 接口）

**作用**：对外提供 REST API，处理 HTTP 请求

**组件**：
- User API：用户相关接口
- Social API：社交相关接口
- IM API：消息相关接口
- IM WS：WebSocket 实时通讯

**职责**：
1. 接收 HTTP 请求
2. 参数验证
3. 调用 RPC 服务
4. 返回 HTTP 响应

**示例**：用户登录流程
```
1. 前端发送 POST /api/user/login
2. User API 接收请求
3. 验证参数（手机号、密码格式）
4. 调用 User RPC 的 Login 方法
5. 返回 Token 给前端
```

#### 第三层：RPC 层（业务逻辑）

**作用**：处理核心业务逻辑，供 API 层和其他服务调用

**组件**：
- User RPC：用户业务逻辑
- Social RPC：社交业务逻辑
- IM RPC：消息业务逻辑

**职责**：
1. 业务逻辑处理
2. 数据库操作
3. 缓存操作
4. 调用其他 RPC 服务

**为什么要分 API 和 RPC？**

```
场景1：前端调用
前端 → User API → User RPC → 数据库

场景2：服务间调用
Social API → User RPC → 数据库
            ↑
         直接调用，不经过 User API
```

**优势**：
- API 层只负责 HTTP 协议处理
- RPC 层专注业务逻辑
- 服务间调用更高效（gRPC 比 HTTP 快）

#### 第四层：数据层

**MySQL**：存储结构化数据
- 用户表（users）
- 好友表（friends）
- 群组表（groups）
- 群成员表（group_members）

**MongoDB**：存储非结构化数据
- 聊天记录（chat_logs）
- 会话记录（conversations）

**Redis**：缓存和临时数据
- 用户 Token
- 在线状态
- 会话缓存

**为什么使用多种数据库？**

```
MySQL：适合结构化数据，支持复杂查询
  ├─ 用户信息（固定字段）
  ├─ 好友关系（需要关联查询）
  └─ 群组信息（需要事务）

MongoDB：适合文档数据，读写性能高
  ├─ 聊天记录（字段可能变化）
  └─ 会话记录（嵌套结构）

Redis：适合缓存，速度极快
  ├─ Token（临时数据）
  ├─ 在线状态（频繁变化）
  └─ 热点数据（减轻数据库压力）
```

#### 第五层：消息队列层

**Kafka**：异步消息处理

**使用场景**：
1. **消息持久化**
   ```
   用户发送消息 → Kafka → Task MQ → MongoDB
                  ↓
               立即返回（不等待入库）
   ```

2. **消息推送**
   ```
   用户发送消息 → Kafka → Task MQ → 推送给接收方
   ```

3. **削峰填谷**
   ```
   高峰期：大量消息 → Kafka 缓存 → 慢慢处理
   低峰期：少量消息 → 直接处理
   ```

#### 第六层：服务治理层

**Etcd**：服务注册与发现

**工作原理**：
```
1. 服务启动时注册到 Etcd
   User RPC 启动 → 注册到 Etcd
   地址：192.168.1.100:8080

2. 其他服务从 Etcd 获取服务地址
   Social API 需要调用 User RPC
   → 从 Etcd 查询 User RPC 地址
   → 调用 192.168.1.100:8080

3. 服务下线时自动从 Etcd 移除
   User RPC 停止 → Etcd 自动移除注册信息
```

**优势**：
- 服务地址动态管理
- 自动故障转移
- 负载均衡

## 3. 技术栈详解

### 3.1 Go-Zero 框架

**什么是 Go-Zero？**

Go-Zero 是一个微服务框架，提供了构建微服务所需的所有工具。

**核心特性**：

1. **API 定义**：使用 .api 文件定义接口
   ```go
   // user.api
   type LoginReq {
       Phone    string `json:"phone"`
       Password string `json:"password"`
   }

   type LoginResp {
       Token  string `json:"token"`
       Expire int64  `json:"expire"`
   }

   @server(
       prefix: /api/user
   )
   service user {
       @handler login
       post /login (LoginReq) returns (LoginResp)
   }
   ```

2. **代码生成**：自动生成代码框架
   ```bash
   goctl api go -api user.api -dir .
   ```
   生成：
   - handler（处理 HTTP 请求）
   - logic（业务逻辑）
   - types（数据结构）

3. **RPC 定义**：使用 .proto 文件定义服务
   ```protobuf
   service User {
       rpc Login(LoginReq) returns(LoginResp);
   }
   ```

4. **服务注册发现**：自动注册到 Etcd
   ```yaml
   Etcd:
     Hosts:
       - 192.168.1.100:2379
     Key: user.rpc
   ```

5. **负载均衡**：自动实现客户端负载均衡

6. **熔断降级**：自动熔断故障服务

### 3.2 gRPC

**什么是 gRPC？**

gRPC 是 Google 开发的高性能 RPC 框架。

**为什么使用 gRPC？**

```
HTTP REST API：
  请求：POST /api/user/login
  数据：JSON 格式
  速度：较慢（文本传输）

gRPC：
  请求：调用 User.Login 方法
  数据：Protocol Buffers（二进制）
  速度：快（二进制传输，体积小）
```

**性能对比**：
```
传输 1000 次用户信息：
HTTP + JSON：  100ms
gRPC + Proto:  20ms  ← 快 5 倍
```

### 3.3 WebSocket

**什么是 WebSocket？**

WebSocket 是一种全双工通信协议，用于实时通讯。

**HTTP vs WebSocket**：

```
HTTP（传统方式）：
  客户端 → 服务器：有新消息吗？
  服务器 → 客户端：没有

  客户端 → 服务器：有新消息吗？（1秒后再问）
  服务器 → 客户端：没有

  客户端 → 服务器：有新消息吗？（1秒后再问）
  服务器 → 客户端：有，这是消息

  问题：需要不断轮询，浪费资源

WebSocket（实时方式）：
  客户端 ←→ 服务器：建立连接

  服务器 → 客户端：有新消息了！（主动推送）
  客户端 → 服务器：收到！

  优势：实时推送，不需要轮询
```

**Easy-Chat 中的使用**：
```
用户 A 发送消息给用户 B：
1. 用户 A 通过 WebSocket 发送消息
2. 服务器收到消息
3. 服务器通过 WebSocket 推送给用户 B
4. 用户 B 实时收到消息
```

### 3.4 Kafka

**什么是 Kafka？**

Kafka 是一个分布式消息队列系统。

**为什么需要消息队列？**

```
没有消息队列：
  用户发送消息 → 直接写入数据库 → 返回成功

  问题：
  - 如果数据库慢，用户等待时间长
  - 如果数据库挂了，消息丢失
  - 高并发时数据库压力大

有消息队列：
  用户发送消息 → 写入 Kafka → 立即返回成功
                    ↓
                 后台慢慢处理
                    ↓
                 写入数据库

  优势：
  - 用户体验好（快速返回）
  - 消息不丢失（Kafka 持久化）
  - 削峰填谷（缓冲高并发）
```

**Easy-Chat 中的使用**：
```
Topic: ws2ms_chat（WebSocket 到消息服务）
  - 用户发送的消息先写入这个 Topic

Topic: ms2ps_chat（消息服务到推送服务）
  - 需要推送的消息写入这个 Topic

Topic: msg_to_mongo（消息持久化）
  - 需要入库的消息写入这个 Topic
```

### 3.5 数据库选型

#### MySQL

**特点**：
- 关系型数据库
- 支持事务（ACID）
- 支持复杂查询（JOIN）

**适用场景**：
```
用户表：
  - 需要保证数据一致性
  - 需要关联查询（如查询用户的所有好友）

好友表：
  - 需要事务（添加好友是双向的）
  - 需要关联查询

群组表：
  - 需要事务（创建群组和添加成员）
  - 需要关联查询
```

#### MongoDB

**特点**：
- 文档数据库
- 无固定结构
- 读写性能高

**适用场景**：
```
聊天记录：
  - 数据量大（百万、千万级）
  - 查询简单（按时间范围查询）
  - 字段可能变化（文本、图片、语音等）

会话记录：
  - 嵌套结构（会话包含最后一条消息）
  - 频繁更新（未读数、最后消息）
```

#### Redis

**特点**：
- 内存数据库
- 速度极快
- 支持多种数据结构

**适用场景**：
```
Token 缓存：
  - 每次请求都要验证 Token
  - 从 Redis 读取比从 MySQL 快 100 倍

在线状态：
  - 频繁变化（用户上线、下线）
  - 不需要持久化

会话缓存：
  - 热点数据（最近的会话）
  - 减轻 MongoDB 压力
```

```go
func (l *RegisterLogic) Register(req *types.RegisterReq) (*types.RegisterResp, error) {
    // 1. 检查手机号是否已注册
    existUser, err := l.svcCtx.UserModel.FindByPhone(l.ctx, req.Phone)
    if err != nil && err != model.ErrNotFound {
        return nil, err
    }
    if existUser != nil {
        return nil, errors.New("手机号已注册")
    }

    // 2. 密码加密
    hashedPassword, err := bcrypt.GenerateFromPassword([]byte(req.Password), bcrypt.DefaultCost)
    if err != nil {
        return nil, err
    }

    // 3. 生成用户 ID
    userId := uuid.New().String()

    // 4. 插入用户表
    user := &model.User{
        Id:       userId,
        Phone:    req.Phone,
        Nickname: req.Nickname,
        Password: string(hashedPassword),
        Avatar:   req.Avatar,
    }
    err = l.svcCtx.UserModel.Insert(l.ctx, user)
    if err != nil {
        return nil, err
    }

    // 5. 生成 JWT Token
    now := time.Now().Unix()
    expire := now + l.svcCtx.Config.Auth.AccessExpire
    token, err := l.getJwtToken(l.svcCtx.Config.Auth.AccessSecret, now, expire, userId)
    if err != nil {
        return nil, err
    }

    // 6. 将 Token 存入 Redis
    err = l.svcCtx.Redis.Setex(fmt.Sprintf("token:%s", userId), token, int(l.svcCtx.Config.Auth.AccessExpire))
    if err != nil {
        return nil, err
    }

    return &types.RegisterResp{
        Token:  token,
        Expire: expire,
    }, nil
}

// 生成 JWT Token
func (l *RegisterLogic) getJwtToken(secretKey string, iat, seconds int64, userId string) (string, error) {
    claims := make(jwt.MapClaims)
    claims["exp"] = iat + seconds
    claims["iat"] = iat
    claims["userId"] = userId
    token := jwt.New(jwt.SigningMethodHS256)
    token.Claims = claims
    return token.SignedString([]byte(secretKey))
}
```

```go
type MsgHandler struct {
    svcCtx *svc.ServiceContext
}

func NewMsgHandler(svcCtx *svc.ServiceContext) *MsgHandler {
    return &MsgHandler{
        svcCtx: svcCtx,
    }
}

// 处理消息
func (h *MsgHandler) Consume(key, val string) error {
    // 1. 解析消息
    var msg types.ChatMsg
    if err := json.Unmarshal([]byte(val), &msg); err != nil {
        return err
    }

    // 2. 保存消息到 MongoDB
    if err := h.saveMsg(&msg); err != nil {
        return err
    }

    // 3. 更新会话
    if err := h.updateConversation(&msg); err != nil {
        return err
    }

    // 4. 推送消息
    if err := h.pushMsg(&msg); err != nil {
        return err
    }

    return nil
}

// 保存消息
func (h *MsgHandler) saveMsg(msg *types.ChatMsg) error {
    chatLog := &model.ChatLog{
        MsgId:            msg.MsgId,
        ConversationType: msg.ConversationType,
        SendId:           msg.SendId,
        RecvId:           msg.RecvId,
        MsgType:          msg.MsgType,
        Content:          msg.Content,
        SendTime:         msg.SendTime,
    }
    return h.svcCtx.ChatLogModel.Insert(context.Background(), chatLog)
}

// 更新会话
func (h *MsgHandler) updateConversation(msg *types.ChatMsg) error {
    // 单聊：更新双方的会话
    if msg.ConversationType == 1 {
        // 更新发送方会话
        h.svcCtx.ConversationModel.UpdateLastMsg(context.Background(), msg.SendId, msg.RecvId, msg)
        // 更新接收方会话（未读数+1）
        h.svcCtx.ConversationModel.UpdateLastMsgAndIncUnread(context.Background(), msg.RecvId, msg.SendId, msg)
    } else {
        // 群聊：更新所有成员的会话
        members, _ := h.svcCtx.GroupMemberModel.FindByGroupId(context.Background(), msg.RecvId)
        for _, member := range members {
            if member.UserId != msg.SendId {
                h.svcCtx.ConversationModel.UpdateLastMsgAndIncUnread(context.Background(), member.UserId, msg.RecvId, msg)
            }
        }
    }
    return nil
}

// 推送消息
func (h *MsgHandler) pushMsg(msg *types.ChatMsg) error {
    // 单聊：推送给接收方
    if msg.ConversationType == 1 {
        return h.svcCtx.WsClient.Push(msg.RecvId, msg)
    } else {
        // 群聊：推送给所有在线成员
        members, _ := h.svcCtx.GroupMemberModel.FindByGroupId(context.Background(), msg.RecvId)
        for _, member := range members {
            if member.UserId != msg.SendId {
                h.svcCtx.WsClient.Push(member.UserId, msg)
            }
        }
    }
    return nil
}
```

```mermaid
        UserAPI-->>前端: 手机号已注册
    else 手机号不存在
```

   - 查询手机号是否已注册
   - 密码加密（使用 bcrypt，不可逆）
   - 生成唯一用户 ID
   - 插入数据库
   - 生成 JWT Token
   - 存储 Token 到 Redis

### 5.4 发送单聊消息流程

```mermaid
sequenceDiagram
    participant 用户A
    participant IMWS
    participant Kafka
    participant TaskMQ
    participant MongoDB
    participant Redis
    participant 用户B

    IMWS->>用户A: 返回ACK确认
    Note over 用户A: {"type":"ack","content":{...}}

    Note over TaskMQ: 异步处理

        TaskMQ->>MongoDB: 保存聊天记录
        Note over MongoDB: chat_logs集合

        TaskMQ->>MongoDB: 更新发送方会话

        Redis-->>TaskMQ: 返回在线状态

            Note over 用户B: 实时收到消息
            用户B->>IMWS: 返回ACK确认
            IMWS->>TaskMQ: 标记消息已送达
        else 用户B离线

            Note over Redis: 用户上线时拉取
        end
    end
```

**详细步骤说明**：

1. **用户 A 发送消息**
   ```json
   {
       "type": "chat",
       "content": {
           "conversationType": 1,
           "recvId": "user_b_id",
           "msgType": 1,
           "content": "你好",
   ```

   - 验证消息格式
   - 生成服务器消息 ID（UUID）
   - 记录发送时间（时间戳）
   - 写入 Kafka（异步处理）
   - 立即返回 ACK（用户体验好）

3. **返回 ACK**
   ```json
   {
       "type": "ack",
       "content": {
           "clientMsgId": "client_msg_123",
           "serverMsgId": "server_msg_456",
           "sendTime": 1678886400,
           "status": "success"
       }
   }
   ```

4. **Task MQ 异步处理**
   - 保存到 MongoDB（持久化）
   - 更新会话（最后消息、未读数）
   - 推送给接收方（如果在线）

5. **推送给用户 B**
   ```json
   {
       "type": "push",
       "content": {
           "msgId": "server_msg_456",
           "conversationType": 1,
           "sendId": "user_a_id",
   ```

```mermaid
sequenceDiagram
    participant 用户A
    participant IMWS
    participant Kafka
    participant TaskMQ
    participant MongoDB
    participant MySQL
    participant Redis
    participant 群成员

    用户A->>IMWS: WebSocket: 发送群消息
    Note over 用户A: {"type":"chat","content":{...}}
    
    IMWS->>IMWS: 验证消息格式
    IMWS->>IMWS: 生成服务器消息ID
    
    IMWS->>Kafka: 写入消息队列
    Note over Kafka: Topic: ws2ms_chat
    Kafka-->>IMWS: 写入成功
    
    IMWS->>用户A: 返回ACK确认
    
    TaskMQ->>Kafka: 消费消息
    
    TaskMQ->>MongoDB: 保存聊天记录
    MongoDB-->>TaskMQ: 保存成功
    
    TaskMQ->>MySQL: 查询群成员列表
    MySQL-->>TaskMQ: 返回成员列表
    Note over TaskMQ: [user_b, user_c, user_d, ...]
    
    loop 遍历每个成员
        alt 成员不是发送者
            TaskMQ->>MongoDB: 更新成员会话(未读数+1)
            MongoDB-->>TaskMQ: 更新成功
            
            TaskMQ->>Redis: 检查成员是否在线
            Redis-->>TaskMQ: 返回在线状态
            
            alt 成员在线
                TaskMQ->>群成员: WebSocket推送消息
                Note over 群成员: 实时收到群消息
            else 成员离线
                TaskMQ->>Redis: 记录离线消息
            end
        end
    end
```

**群聊与单聊的区别**：

| 特性 | 单聊 | 群聊 |
|------|------|------|
| 接收方 | 1个用户 | 多个用户 |
| 会话更新 | 更新2个会话（发送方、接收方） | 更新N个会话（所有群成员） |
| 消息推送 | 推送给1个用户 | 推送给N个用户 |
| 未读数 | 接收方未读数+1 | 所有成员未读数+1（除发送者） |
| 性能考虑 | 简单 | 需要优化（批量操作） |

```mermaid
            WS->>R: 删除在线状态
            Note right of WS: DEL online:userId
        else 网络异常断开
            Note over F,WS: 连接断开
            F->>F: 检测到连接断开
            F->>F: 启动重连机制
            
            loop 重连循环
                F->>WS: 尝试重新连接
                alt 重连成功
                    WS->>R: 重新注册在线状态
                    WS->>F: 连接成功
                    F->>F: 恢复心跳
                else 重连失败
                    F->>F: 等待5秒后重试
                end
            end
        else 心跳超时
            Note over WS,R: 超过5分钟没有心跳
            R->>R: TTL过期,自动删除在线状态
            Note right of R: 用户被标记为离线
        end
    end
```

**WebSocket 重连策略**：

```javascript
// 前端重连逻辑
class WebSocketClient {
    constructor(url, token) {
        this.url = url;
        this.token = token;
        this.ws = null;
        this.reconnectAttempts = 0;
        this.maxReconnectAttempts = 10;
        this.reconnectInterval = 5000; // 5秒
        this.heartbeatInterval = 30000; // 30秒
        this.heartbeatTimer = null;
    }

    connect() {
        this.ws = new WebSocket(`${this.url}?token=${this.token}`);
        
        this.ws.onopen = () => {
            console.log('WebSocket连接成功');
            this.reconnectAttempts = 0;
            this.startHeartbeat();
        };
        
        this.ws.onclose = (event) => {
            console.log('WebSocket连接关闭', event.code, event.reason);
            this.stopHeartbeat();
            
            if (event.code === 4001) {
                // Token无效,跳转到登录页面
                window.location.href = '/login';
            } else {
                // 网络异常,尝试重连
                this.reconnect();
            }
        };
        
        this.ws.onerror = (error) => {
            console.error('WebSocket错误', error);
        };
        
        this.ws.onmessage = (event) => {
            const msg = JSON.parse(event.data);
            this.handleMessage(msg);
        };
    }

    reconnect() {
        if (this.reconnectAttempts >= this.maxReconnectAttempts) {
            console.error('重连次数超过限制');
            return;
        }
        
        this.reconnectAttempts++;
        console.log(`尝试重连 (${this.reconnectAttempts}/${this.maxReconnectAttempts})`);
        
        setTimeout(() => {
            this.connect();
        }, this.reconnectInterval);
    }

    startHeartbeat() {
        this.heartbeatTimer = setInterval(() => {
            if (this.ws.readyState === WebSocket.OPEN) {
                this.send({ type: 'heartbeat' });
            }
        }, this.heartbeatInterval);
    }

    stopHeartbeat() {
        if (this.heartbeatTimer) {
            clearInterval(this.heartbeatTimer);
            this.heartbeatTimer = null;
        }
    }

    send(data) {
        if (this.ws.readyState === WebSocket.OPEN) {
            this.ws.send(JSON.stringify(data));
        }
    }

    handleMessage(msg) {
        switch (msg.type) {
            case 'heartbeat':
                // 心跳响应
                break;
            case 'push':
                // 收到新消息
                this.onNewMessage(msg.content);
                break;
            case 'notify':
                // 收到通知
                this.onNotify(msg.content);
                break;
        }
    }
}
```

#### 8.4.1 MySQL 查询优化

**1. 使用索引**

```sql

SELECT * FROM users WHERE nickname LIKE '%张三%';

SELECT * FROM users WHERE nickname LIKE '张三%';  -- 前缀匹配可以使用索引

ALTER TABLE users ADD FULLTEXT INDEX ft_nickname (nickname);
```

```sql

SELECT * FROM users WHERE id = 'user_a';

SELECT id, nickname, avatar FROM users WHERE id = 'user_a';
```

```sql
-- 不推荐(查询所有数据)
SELECT * FROM users ORDER BY create_time DESC;

-- 推荐(分页查询)
SELECT * FROM users ORDER BY create_time DESC LIMIT 0, 20;
```

**4. 使用JOIN代替子查询**

```sql
-- 不推荐(子查询)
SELECT * FROM friends
WHERE friend_id IN (SELECT id FROM users WHERE status = 1);

-- 推荐(JOIN)
SELECT f.* FROM friends f
INNER JOIN users u ON f.friend_id = u.id
```

**2. 使用投影(只查询需要的字段)**

```javascript

// 推荐
db.chat_logs.find(
    { msgId: "msg_001" },
    { msgId: 1, content: 1, sendTime: 1, _id: 0 }
);
```

**3. 使用聚合管道优化复杂查询**

```javascript
// 统计每个用户的消息数量
db.chat_logs.aggregate([
    { $match: { conversationType: 2, recvId: "group_001" } },
    { $group: { _id: "$sendId", count: { $sum: 1 } } },
    { $sort: { count: -1 } },
    { $limit: 10 }
]);
```

#### 8.4.3 Redis 优化

**1. 使用Pipeline批量操作**

```go
// 不推荐(多次网络请求)
for _, userId := range userIds {
    redis.Get(ctx, fmt.Sprintf("online:%s", userId))
}

// 推荐(Pipeline批量操作)
pipe := redis.Pipeline()
for _, userId := range userIds {
    pipe.Get(ctx, fmt.Sprintf("online:%s", userId))
}
results, _ := pipe.Exec(ctx)
```

**2. 设置合理的过期时间**

```redis
# 不推荐(永不过期,占用内存)

```

**3. 使用Hash代替多个Key**


## 10. 常见问题解答

### 10.1 如何保证消息不丢失？

**问题**：在分布式系统中，消息可能在多个环节丢失，如何保证消息不丢失？

**解决方案**：

#### 1. 客户端层面

```javascript
// 前端消息发送机制
class MessageSender {
    constructor() {
        this.pendingMessages = [];  // 待发送消息队列
        this.sentMessages = [];     // 已发送但未确认的消息
    }

    // 发送消息
    async sendMessage(message) {
        // 1. 生成客户端消息ID
        message.clientMsgId = this.generateMsgId();
        
        // 2. 保存到本地数据库
        await this.saveToLocalDB(message);
        
        // 3. 添加到待发送队列
        this.pendingMessages.push(message);
        
        // 4. 发送消息
        this.send(message);
    }

    // 发送消息到服务器
    send(message) {
        if (this.ws.readyState === WebSocket.OPEN) {
            this.ws.send(JSON.stringify({
                type: 'chat',
                content: message
            }));
            
            // 移到已发送队列
            this.sentMessages.push(message);
            
            // 设置超时重发
            setTimeout(() => {
                if (!message.acked) {
                    console.log('消息未确认,重新发送:', message.clientMsgId);
                    this.send(message);
                }
            }, 5000);  // 5秒后重发
        } else {
            // WebSocket未连接,保持在待发送队列
            console.log('WebSocket未连接,消息保存在队列中');
        }
    }

    // 收到ACK确认
    onAck(ack) {
        const message = this.sentMessages.find(m => m.clientMsgId === ack.clientMsgId);
        if (message) {
            message.acked = true;
            message.serverMsgId = ack.serverMsgId;
            message.sendTime = ack.sendTime;
            
            // 更新本地数据库
            this.updateLocalDB(message);
            
            // 从已发送队列移除
            this.sentMessages = this.sentMessages.filter(m => m.clientMsgId !== ack.clientMsgId);
        }
    }

    // WebSocket重连后,重发未确认的消息
    onReconnect() {
        console.log('WebSocket重连,重发未确认的消息');
        this.sentMessages.forEach(message => {
            this.send(message);
        });
    }
}
```

#### 2. 服务器层面

```go
// IM WebSocket 服务
func (h *WebSocketHandler) HandleMessage(conn *websocket.Conn, msg *types.WsMsg) error {
    // 1. 验证消息格式
    if err := h.validateMessage(msg); err != nil {
        return err
    }

    // 2. 检查消息是否重复(根据clientMsgId)
    exists, err := h.redis.Exists(ctx, fmt.Sprintf("msg_dedup:%s", msg.Content.ClientMsgId)).Result()
    if err != nil {
        return err
    }
    if exists > 0 {
        // 消息已处理,直接返回ACK
        return h.sendAck(conn, msg.Content.ClientMsgId, msg.Content.ServerMsgId)
    }

    // 3. 生成服务器消息ID
    serverMsgId := uuid.New().String()
    msg.Content.ServerMsgId = serverMsgId
    msg.Content.SendTime = time.Now().Unix()

    // 4. 写入Kafka(持久化)
    err = h.kafka.Send(ctx, "ws2ms_chat", serverMsgId, msg.Content)
    if err != nil {
        // Kafka写入失败,返回错误
        return err
    }

    // 5. 设置去重标记(24小时过期)
    h.redis.Set(ctx, fmt.Sprintf("msg_dedup:%s", msg.Content.ClientMsgId), serverMsgId, 24*time.Hour)

    // 6. 返回ACK
    return h.sendAck(conn, msg.Content.ClientMsgId, serverMsgId)
}
```

```go
    // 1. 解析消息
    var msg types.ChatMsg
    if err := json.Unmarshal([]byte(val), &msg); err != nil {
        return err
    }

    // 2. 检查消息是否已处理(幂等性)

    if err == nil && exists != nil {

    if err != nil {
        return err

        if err := h.saveMsg(sc, &msg); err != nil {
            return err
        }

        // 3.2 更新会话
        if err := h.updateConversation(sc, &msg); err != nil {
            return err
        }

        // 3.3 推送消息
        if err := h.pushMsg(sc, &msg); err != nil {
            // 推送失败不影响消息保存
            log.Error("推送消息失败:", err)
        }

        return nil
    })

    if err != nil {
        // 处理失败,返回错误,Kafka会重新投递
        return err
    }

    return nil
```

#### 2. ACK 确认机制

```javascript
// 前端消息状态管理
class MessageStatusManager {
    // 发送消息
    async sendMessage(message) {
        // 1. 消息状态设置为"发送中"
        message.status = 'sending';
        this.updateUI(message);

        // 2. 发送到服务器
        this.ws.send(JSON.stringify({
            type: 'chat',
            content: message
        }));

        // 3. 等待服务器ACK
        const ack = await this.waitForAck(message.clientMsgId, 5000);
        if (ack) {
            // 收到ACK,状态更新为"已发送"
            message.status = 'sent';
            message.serverMsgId = ack.serverMsgId;
            this.updateUI(message);
        } else {
            // 超时未收到ACK,状态更新为"发送失败"
            message.status = 'failed';
            this.updateUI(message);
        }
    }

    // 收到送达回执
    onDelivered(msgId) {
        const message = this.findMessage(msgId);
        if (message) {
            message.status = 'delivered';
            this.updateUI(message);
        }
    }

    // 收到已读回执
    onRead(msgId) {
        const message = this.findMessage(msgId);
        if (message) {
            message.status = 'read';
            this.updateUI(message);
        }
    }
}
```

```go
// 推送消息
func (h *PushHandler) PushMsg(msg *types.ChatMsg) error {
    // 1. 查询接收方在线状态
    online, err := h.redis.Get(ctx, fmt.Sprintf("online:%s", msg.RecvId)).Result()
    if err == redis.Nil {
        // 用户离线,存储离线消息
        return h.saveOfflineMsg(msg)
    }

    // 2. 用户在线,推送消息
    err = h.wsClient.Push(msg.RecvId, msg)
    if err != nil {
        // 推送失败,存储离线消息
        return h.saveOfflineMsg(msg)
    }

    // 3. 等待接收方ACK(5秒超时)
    ack, err := h.waitForAck(msg.MsgId, 5*time.Second)
    if err != nil {
        // 未收到ACK,存储离线消息
        return h.saveOfflineMsg(msg)
    }

    // 4. 收到ACK,更新消息状态为"已送达"
    return h.updateMsgStatus(msg.MsgId, "delivered")
}

// 保存离线消息
func (h *PushHandler) saveOfflineMsg(msg *types.ChatMsg) error {
    data, _ := json.Marshal(msg)
    return h.redis.LPush(ctx, fmt.Sprintf("offline_msg:%s", msg.RecvId), data).Err()
}

// 用户上线时拉取离线消息
func (h *PushHandler) PullOfflineMsg(userId string) ([]*types.ChatMsg, error) {
    // 1. 获取所有离线消息
    data, err := h.redis.LRange(ctx, fmt.Sprintf("offline_msg:%s", userId), 0, -1).Result()
    if err != nil {
        return nil, err
    }

    // 2. 解析消息
    var messages []*types.ChatMsg
    for _, d := range data {
        var msg types.ChatMsg
        json.Unmarshal([]byte(d), &msg)
        messages = append(messages, &msg)
    }

    // 3. 删除离线消息
    h.redis.Del(ctx, fmt.Sprintf("offline_msg:%s", userId))

    return messages, nil
}
```
